## 冲突处理结论

`gateway/run.py` 的冲突来自上游在 `v2026.6.19` 中把 kanban watcher 从
大文件抽到了 `gateway/kanban_watchers.py`。本地旧提交修改的是旧的内联
watcher，因此不能把旧 block 放回 `gateway/run.py`。最终处理方式是保留
上游抽取后的结构，并把本地 `delivery_mode == "agent"` 的分流、事件 claim
范围和 agent delivery 注入逻辑迁移到 `gateway/kanban_watchers.py`。

`tests/agent/test_bedrock_adapter.py` 的冲突来自上游新增 streaming IAM fallback
测试，同时本地提交把 runtime client cache key 改成可区分 Bedrock bearer token
的形式。最终保留上游新增测试，并让手动写入 cache 的测试统一使用
`_runtime_client_cache_key()`，避免被 `AWS_BEARER_TOKEN_BEDROCK` 环境变量影响。

`tests/test_tui_gateway_server.py` 的冲突只是测试插入位置重叠。最终同时保留
上游 profile/SSH cwd 测试和本地 TUI delivery identity 测试。

## 仍需重点验证的能力

- Bedrock bearer token 下的 native Converse / auxiliary client 路径。
- Gateway kanban notification 与 `delivery_mode == "agent"` 分流。
- TUI/Desktop kanban delivery poller 的完成、忙碌回退和 session finalize 停止逻辑。
